# 宝塔公网部署版设计

## 目标

把现有 PDF 图片提取工具整理成适合通过宝塔面板部署到独立公网子域名的版本，重点解决服务器 fresh install、运行配置、健康检查和公开访问下的基础稳定性问题。

## 部署前提

- 部署形态为独立子域名，例如 `pdf.example.com`
- 站点通过宝塔面板里的 `Nginx` 对外提供 HTTPS
- 应用进程运行在同一台服务器上，仅监听本机端口，例如 `127.0.0.1:18081`
- 第一版允许公开访问，不引入账号系统

## 设计结论

### 1. 运行架构

保持单体 FastAPI 应用，不引入数据库、消息队列或额外前端构建链。对外仍是一个页面加一个上传提取接口，但由宝塔里的反向代理统一处理域名、HTTPS、上传大小和超时。

### 2. 打包与安装

修正 `pyproject.toml` 的包发现配置，避免仓库中的 `tmp`、`output` 等目录影响安装；同时明确把模板文件打进包里，保证服务器上通过标准 `pip install -e .` 或 `pip install .` 都能正常运行。

### 3. 配置方式

把当前写死在代码里的服务限制改为环境变量驱动，并提供安全默认值。第一版至少支持：

- `PIC_EXTRACTOR_MAX_FILE_SIZE_MB`
- `PIC_EXTRACTOR_MAX_PAGES`

这样既便于本地调试，也便于宝塔启动脚本统一管理。

### 4. 请求处理

保持同步提取逻辑本身不变，但把提取和打包放进线程池调用，避免在 FastAPI 的异步路由中直接阻塞事件循环。这样不改变业务结果，却能让公开访问时的并发退化更可控。

### 5. 运维可见性

新增一个轻量健康检查接口，例如 `/healthz`，用于宝塔反代或人工探活。第一版不引入复杂监控，只提供明确的“服务是否启动”判断入口。

### 6. 文档与部署指引

在 README 中新增宝塔部署章节，写清：

- Python 环境和依赖安装
- 启动命令
- 反向代理目标端口
- 建议的 `Nginx` 上传大小和超时
- 独立子域名部署方式

## 实现范围

本次覆盖：

- Python 包安装与模板打包修复
- 环境变量配置
- 健康检查接口
- 线程池执行提取任务
- 宝塔部署文档
- 对应测试补充

本次不覆盖：

- 账号系统或访问控制
- Redis / Celery / 异步任务队列
- 数据库存储
- 多实例调度
- 应用层复杂限流

## 风险与取舍

- 公开上传 PDF 仍然存在 CPU 和内存压力，第一版主要依赖文件大小、页数限制和 Nginx 超时来控制风险。
- 线程池只能减少事件循环阻塞，不能把重任务变成真正的后台任务；但对于单机小流量公开工具站，这个权衡足够务实。
- 由于采用独立子域名，本次不优先处理主站子路径挂载的兼容问题，避免为当前部署方式引入额外复杂度。
